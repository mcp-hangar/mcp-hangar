## Overview

Stream live stdout/stderr from running provider subprocesses and containers to browser clients over WebSocket. Providers previously crashed silently -- engineers had no visibility into what a provider was doing. v4.0 fixes four critical gaps: live stderr reading (not post-mortem drain), Docker DEVNULL-to-PIPE, per-provider ring buffer for history, and a WebSocket endpoint plus UI log viewer in the provider detail page.

## Milestone Summary

**Key Decisions:**

- `DEFAULT_MAX_LINES = 1000` per provider -- configurable at construction time
- `on_append` callback invoked outside `ProviderLogBuffer._lock` -- prevents I/O under lock per CLAUDE.md
- `get_or_create_log_buffer` uses `_registry_lock` for thread-safe idempotency
- Lazy imports in `init_log_buffers()` to break circular dependency between server and infrastructure layers
- `DockerLauncher` changed from `stderr=subprocess.DEVNULL` to `stderr=subprocess.PIPE` -- critical for live capture
- BLE001 fault-barrier in stderr reader thread -- pipe errors swallowed, thread never crashes on process kill
- Broadcaster callbacks are per-provider async callables -- enables concurrent streaming to multiple clients
- `on_append` invoked outside `ProviderLogBuffer._lock` per CLAUDE.md no-I/O-under-lock rule
- Deferred buffer injection: `Provider` constructed first, buffer injected after by `init_log_buffers()`
- `LogStreamBroadcaster` singleton carried on `ApplicationContext` for WebSocket endpoint access
- Amber for stderr, gray for stdout -- conventional terminal color coding matching terminal standards
- `useProviderLogs` follows same auto-reconnect pattern as `useWebSocket` hook -- consistent

**Issues Resolved:**

- Providers crashed silently with no runtime output visibility -- all stderr now captured to ring buffer
- `DockerLauncher` discarded stderr via `DEVNULL` -- live capture impossible before this change

**Issues Deferred:**

- `stdout` capture: only `stderr` captured in v4.0; stdout capture for subprocesses would require a different pipe routing approach
- Log persistence across restarts: ring buffers are in-memory only, lost on provider restart

**Technical Debt Incurred:**

- None: all implementation is consistent with existing patterns (CQRS, domain contracts, no-I/O-under-lock)
