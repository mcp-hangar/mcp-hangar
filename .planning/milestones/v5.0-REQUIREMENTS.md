# Requirements: v5.0 Platform Management Console

**Defined:** 2026-03-22
**Status:** PLANNING
**Core Value:** Transform MCP Hangar from a config-file-driven system into a full platform management console -- CRUD for providers/groups, discovery source management, MCP catalog, RBAC management, tool access policy editing, and config export/backup -- all from the UI, API, and static config.

## Requirements

### Provider & Group CRUD (C1/U1)

- [x] **CRUD-01**: `CreateProviderCommand`, `UpdateProviderCommand`, `DeleteProviderCommand` exist as frozen CQRS commands. Handlers create/update/delete `Provider` aggregates in `PROVIDERS` state and emit `ProviderRegistered`, `ProviderUpdated`, `ProviderDeregistered` domain events (all new -- `ProviderRegistered` is distinct from the existing `ProviderDiscovered` discovery event; includes `source: str` field to distinguish `"api"` / `"config"` / `"discovery"` origin). REST endpoints `POST /api/providers`, `PUT /api/providers/{id}`, `DELETE /api/providers/{id}` route through CQRS dispatch. Deleting a running provider stops it first.

- [ ] **CRUD-02**: `CreateGroupCommand`, `UpdateGroupCommand`, `DeleteGroupCommand`, `AddGroupMemberCommand`, `RemoveGroupMemberCommand` exist. Handlers create/update/delete `ProviderGroup` aggregates in `GROUPS` state and emit domain events. Reuse existing `GroupCreated`, `GroupMemberAdded`, `GroupMemberRemoved` events; add new `GroupUpdated`, `GroupDeleted` events following the existing `Group*` naming convention (not `ProviderGroup*`). REST endpoints `POST /api/groups`, `PUT /api/groups/{id}`, `DELETE /api/groups/{id}`, `POST /api/groups/{id}/members`, `DELETE /api/groups/{id}/members/{member_id}`.

- [x] **CRUD-03**: Config serializer module (`server/config_serializer.py`) with `serialize_providers()`, `serialize_groups()`, `serialize_full_config()`. `write_config_backup(path)` writes current in-memory state to YAML with rotation (config.yaml.bak1..bak5, oldest deleted). REST endpoints `POST /api/config/export` (returns YAML string), `POST /api/config/backup` (writes backup file, returns path).

- [ ] **CRUD-04**: `ProviderCreateDrawer` and `ProviderEditDrawer` React components (multi-step: mode -> connection -> behavior -> access). `ProviderDeleteDialog` with state warning. `ProvidersPage` has "+New Provider" button and Edit/Delete row actions. MSW handlers for C1 endpoints.

- [ ] **CRUD-05**: `GroupCreateDrawer` and `GroupEditDrawer` React components (strategy, circuit breaker, health config). `GroupMemberPanel` (add/remove members, weight/priority, per-member tool access). `GroupsPage` has "+New Group" button and member management in detail panel. MSW handlers for group CRUD endpoints.

### Discovery Source Management (C2/U2)

- [ ] **DISC-01**: `DiscoveryRegistry` class with `register_source()`, `deregister_source()`, `update_source()`, `list_sources()`, `get_source()`, `enable_source()`, `disable_source()` methods. `DiscoverySourceSpec` value object (source_type, config dict, enabled flag, mode). Sources can be added/removed/updated at runtime without restart.

- [ ] **DISC-02**: REST endpoints `POST /api/discovery/sources` (register), `PUT /api/discovery/sources/{id}` (update), `DELETE /api/discovery/sources/{id}` (deregister), `POST /api/discovery/sources/{id}/scan` (trigger immediate scan), `PUT /api/discovery/sources/{id}/enable` (toggle). All route through CQRS commands.

- [ ] **DISC-03**: `AddDiscoverySourceWizard` React component (multi-step: type picker -> config form -> test -> confirm). `DiscoveryPage` gains enabled/disabled toggle, edit/remove source, and "+Add Source" button. MSW handlers for C2 discovery endpoints.

### Static MCP Catalog (C2/U2)

- [ ] **CAT-01**: `McpProviderEntry` domain model (entry_id, name, description, mode, command, image, tags, verified, source, required_env, builtin: bool). `McpCatalogRepository` interface with SQLite implementation. Built-in YAML seed file (`data/catalog_seed.yaml`) loaded on bootstrap if catalog table is empty; seed entries marked `builtin=True` (cannot be deleted via API, only custom entries can).

- [ ] **CAT-02**: REST endpoints `GET /api/catalog` (list with search/tag filter), `GET /api/catalog/{id}` (detail), `POST /api/catalog/entries` (add custom), `DELETE /api/catalog/entries/{id}` (remove custom), `POST /api/catalog/{id}/deploy` (creates a provider from the catalog entry). Deploy emits `ProviderRegistered` event.

- [ ] **CAT-03**: `CatalogPage` React component (new route `/catalog`): search bar, tag filters, grid of `McpProviderEntry` cards, Deploy button. `CatalogEntryDrawer` (detail slide-over with deploy confirmation). Sidebar gains "Catalog" link. MSW handlers for catalog endpoints.

### RBAC Management (C3/U3)

- [ ] **RBAC-01**: `RoleStore` (SQLite) with `create_role()`, `update_role()`, `delete_role()`, `get_role()`, `list_roles()`. Built-in roles are immutable (cannot update/delete). Custom roles stored in SQLite with permissions as JSON array. Domain events `RoleCreated`, `RoleUpdated`, `RoleDeleted`.

- [ ] **RBAC-02**: REST endpoints `GET /api/roles` (list all, builtin flag), `POST /api/roles` (create custom), `PUT /api/roles/{name}` (update custom), `DELETE /api/roles/{name}` (delete custom). `GET /api/principals` (list known principals with role counts). `POST /api/principals/{id}/roles` (assign role), `DELETE /api/principals/{id}/roles/{role}` (revoke role). `GET /api/principals/{id}/permissions` (effective permissions, resolved through role hierarchy).

- [ ] **RBAC-03**: `SecurityPage` refactored into tabs: Events | Roles | Principals. Roles tab: table with builtin badge, "+Create Role" drawer with permission picker (checkbox grid by resource). Principals tab: table with role assignment dropdown, effective permissions expand. MSW handlers for C3 endpoints.

### Tool Access Policy CRUD (C3/U3)

- [ ] **TAP-01**: REST endpoints for tool access policy management: `GET /api/providers/{id}/access`, `PUT /api/providers/{id}/access`, `DELETE /api/providers/{id}/access` (provider-level). `GET /api/groups/{id}/access`, `PUT /api/groups/{id}/access`, `DELETE /api/groups/{id}/access` (group-level). `GET /api/groups/{id}/members/{mid}/access`, `PUT /api/groups/{id}/members/{mid}/access`, `DELETE /api/groups/{id}/members/{mid}/access` (member-level). Domain event `ToolAccessPolicyUpdated` emitted on changes.

- [ ] **TAP-02**: `ToolAccessPolicyEditor` shared React component (tag-input with autocomplete for allow/deny patterns). Used in `ProviderDetailPage` (new "Access" tab), `GroupsPage` (group-level + per-member editors). MSW handlers for access policy endpoints.

### Config Export & Diff (C4/U4)

- [ ] **CFG-01**: `serialize_full_config()` extended to include all sections: providers, groups, discovery.sources, catalog.custom_entries, security.custom_roles. `GET /api/config/diff` returns structured diff (added/removed/changed keys) comparing in-memory state vs on-disk config file. Full backup snapshot includes all sections.

- [ ] **CFG-02**: `ConfigPage` refactored with tabs: Current Config | Export/Backup | Diff. Export tab: POST button -> YAML viewer with copy button. Backup tab: POST button -> success toast with file path. Diff tab: colored diff view (green for added, red for removed, yellow for changed). Radix Toast notification system added for all mutation feedback.

- [ ] **CFG-03**: Round-trip integrity: `serialize_full_config() -> write YAML -> load_config_from_file() -> serialize_full_config()` produces identical output (comparison is between two `serialize()` outputs, both post-normalization -- default values like `idle_ttl_s: 300` are always included). Backup rotation (config.yaml.bak1..bak5) preserves last 5 backups. Integration tests verify round-trip and rotation.

### Cross-Cutting

- [ ] **XC-01**: All existing v0.9-v4.0 functionality preserved. No regressions in existing tests. Existing config files continue to work without modification.

- [ ] **XC-02**: TypeScript type definitions created for all new API contracts: `catalog.ts`, `roles.ts`, `access.ts` in `src/types/`. Complete MSW handlers for C1-C4 endpoints. Skeleton loaders and consistent form validation across all new forms.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Drag-and-drop provider ordering | Nice-to-have, not table-stakes |
| Catalog marketplace (publish/share) | Internal catalog sufficient for v5.0 |
| OIDC/SSO login flow | Auth is configured server-side, not end-user login |
| Real-time collaborative editing | Single-admin editing sufficient |
| Config version history (git-like) | Backup rotation provides sufficient safety |
| Undo/redo for CRUD operations | Standard delete confirmation sufficient |
| Bulk import/export of providers | Single CRUD + catalog deploy covers use cases |
| Mobile responsive CRUD forms | Desktop-focused management UI |
| Optimistic concurrency (ETag/If-Match) on REST CRUD | Single-admin assumption; last-write-wins acceptable for v5.0. Event sourcing provides audit trail if conflicts occur. |
