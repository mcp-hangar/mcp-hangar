# Milestone v8.0: Behavioral Profiling Alpha

**Status:** IN PROGRESS (4/6 phases complete)
**Target:** 2026-05-15
**PyPI release:** v0.14.0
**Phases:** 42-47
**Track:** Core (Python), Enterprise (BSL)
**License:** BSL 1.1 -- all new code goes in `enterprise/behavioral/`

## Overview

Know what your MCP servers actually do -- not what they say they do. This milestone builds the behavioral profiling stack: per-container network connection logging, baseline storage, deviation detection, and tool schema drift tracking. It also closes the loop on v7.0 by adding dashboard auth enforcement and the license key infrastructure required to activate enterprise modules.

All behavioral profiling code is developed under BSL 1.1 in `enterprise/behavioral/`. The core defines contracts (`IBehavioralProfiler`, `IBaselineStore`) in `domain/contracts/` (MIT). Enterprise implements them.

OTEL P1 items from v6.0 (Phases 33-34) land in this same release.

## Phases

### Phase 42: Behavioral Profiling Contracts + Core Infrastructure (BSL groundwork)

**Goal:** Define the MIT-licensed contracts that behavioral profiling enterprise code implements. Set up the learning/enforcing mode state machine. No behavioral intelligence yet -- just the plumbing.
**Depends on:** Phase 37 (enterprise/ directory and import boundary established)
**Target:** v0.14.0
**License:** MIT (contracts in `domain/contracts/`), BSL (`enterprise/behavioral/` infrastructure)
**Plans:** 3 plans

Plans:
- [x] 42-01-PLAN.md -- MIT: `domain/contracts/behavioral.py`: `IBehavioralProfiler`, `IBaselineStore`, `IDeviationDetector`. `BehavioralMode` enum: `LEARNING | ENFORCING | DISABLED`. `NetworkObservation` value object: timestamp, provider_id, destination_host, destination_port, protocol, direction.
- [x] 42-02-PLAN.md -- BSL: `enterprise/behavioral/baseline_store.py`: SQLite-backed `BaselineStore` implementing `IBaselineStore`. Schema: `provider_observations(provider_id, host, port, protocol, first_seen, last_seen, observation_count)`.
- [x] 42-03-PLAN.md -- BSL: `enterprise/behavioral/__init__.py` + bootstrap wiring in `enterprise/behavioral/bootstrap.py`. Tests: `BaselineStore` CRUD, `BehavioralMode` transitions.

**Success Criteria:**
1. `IBehavioralProfiler`, `IBaselineStore`, `IDeviationDetector` defined in `domain/contracts/`.
2. `BaselineStore` stores and retrieves `NetworkObservation` records in SQLite.
3. `NullBehavioralProfiler` in core: no-op, used when enterprise modules not loaded.
4. Enterprise behavioral module loads when `HANGAR_LICENSE_KEY` is present.

**Depends on:** Phase 42
**Target:** v0.14.0
**License:** BSL (`enterprise/behavioral/network_monitor.py`)

Plans:
- [x] 43-01-PLAN.md -- BSL: Docker network monitor + /proc/net/tcp parser + container label injection. DockerNetworkMonitor with exec_run ss/proc fallback, pure parsers, --label mcp-hangar.provider-id in Docker/Container launchers.
- [x] 43-02-PLAN.md -- BSL: K8s network monitor with NetworkPolicy audit event watching + pod exec fallback. K8sNetworkMonitor using kubernetes SDK.
- [x] 43-03-PLAN.md -- BSL: ConnectionLogWorker background worker + bootstrap wiring. Daemon thread orchestrating monitors, config parsing (behavioral.connection_logging.*), server lifecycle integration.

**Success Criteria:**
1. Docker provider: outbound connections to new destinations produce `NetworkObservation` records in SQLite.
2. K8s provider: NetworkPolicy audit log events produce `NetworkObservation` records.
3. LEARNING mode: all observations stored without alerting.

### Phase 44: Behavioral Deviation Detection (P0)

**Goal:** Compare runtime network behavior against stored baseline. Alert on new destinations, unusual frequency, or unexpected protocols. Emit `BehavioralDeviationDetected` domain events.
**Depends on:** Phase 43
**Target:** v0.14.0
**License:** BSL (`enterprise/behavioral/deviation_detector.py`)
**Plans:** 3 plans

Plans:
- [x] 44-01-PLAN.md -- MIT domain types (DeviationType enum, BehavioralDeviationDetected event, OTEL conventions, Prometheus counter) + BSL DeviationDetector with 3 detection rules + unit tests
- [x] 44-02-PLAN.md -- Profiler ENFORCING refactoring + BehavioralDeviationEventHandler (OTLP + Prometheus bridge) + bootstrap wiring + integration tests
- [x] 44-03-PLAN.md -- Config parsing validation + end-to-end SC44-1 through SC44-4 verification tests + regression check

**Success Criteria:**
1. Provider connecting to a destination not in its baseline produces `BehavioralDeviationDetected` event within 30 seconds.
2. Deviation events exported through OTLP with governance attributes.
3. `mcp_hangar_behavioral_deviations_total` metric increments.
4. Thresholds configurable per-provider in config YAML (`behavioral.frequency_threshold_multiplier`).

### Phase 45: Tool Schema Drift Detection (P0)

**Goal:** Store tool schema fingerprints at provider startup. On subsequent startups, compare against stored fingerprints. Alert on additions, removals, or parameter changes.
**Depends on:** Phase 44
**Target:** v0.14.0
**License:** BSL (`enterprise/behavioral/schema_tracker.py`)
**Plans:** 3 plans

Plans:
- [x] 45-01-PLAN.md -- BSL SchemaTracker class with SQLite storage + compute_schema_hash + bootstrap wiring (conditional enterprise loading)
- [x] 45-02-PLAN.md -- MIT SchemaChangeType enum + ToolSchemaChanged event + Provider.get_tool_schemas() + TOOL_SCHEMA_DRIFTS_TOTAL counter + ToolSchemaChangeHandler + bootstrap wiring
- [x] 45-03-PLAN.md -- Tests: 30 tests for SchemaTracker (hash, first-seen, ADDED, REMOVED, MODIFIED, mixed, isolation) + handler (SC45-1-4, error handling, metrics)

**Success Criteria:**
1. Provider restarting with a new tool emits `ToolSchemaChanged(change_type: ADDED)`.
2. Provider restarting with a removed tool emits `ToolSchemaChanged(change_type: REMOVED)`.
3. Provider restarting with modified tool parameter schema emits `ToolSchemaChanged(change_type: MODIFIED)`.
4. First-seen schemas are stored without alerting (no baseline to compare against).

### Phase 46: Behavioral Report Export + Resource Profiling (P1)

**Goal:** Generate per-provider behavioral reports (observed destinations, tool usage, resource consumption). Add basic resource usage tracking (CPU/memory/network I/O per container).
**Depends on:** Phase 45
**Target:** v0.14.0

**Plans:** 3 plans

Plans:

### Phase 47: License Key Infrastructure + Dashboard Auth Enforcement

**Goal:** Implement license key validation so enterprise modules activate conditionally. Enforce authentication on all dashboard routes -- no anonymous access.
**Depends on:** Phase 36 (enterprise bootstrap), Phase 42 (enterprise module pattern established)
**Target:** v0.14.0
**License:** BSL (`enterprise/auth/license.py`), MIT (bootstrap gate in `server/bootstrap/`)
**Plans:** 3 plans

Plans:
- [ ] 47-01-PLAN.md -- BSL: `LicenseValidator`: validates `HANGAR_LICENSE_KEY` env var (format check + optional online verification). Returns `LicenseTier` enum: `COMMUNITY | PRO | ENTERPRISE`. Bootstrap uses tier to decide which enterprise modules to load.
- [ ] 47-02-PLAN.md -- MIT: update `server/bootstrap/__init__.py`: if `HANGAR_LICENSE_KEY` absent or invalid, load null implementations. If valid, load enterprise modules. Log tier at startup.
- [ ] 47-03-PLAN.md -- Dashboard auth enforcement: all React routes require authenticated session. Backend middleware rejects unauthenticated requests to dashboard API with 401. Tests: unauthenticated request to dashboard returns 401; authenticated request succeeds.

**Success Criteria:**
1. Server starts without `HANGAR_LICENSE_KEY` using null enterprise implementations. Core features (lifecycle, health, metrics) fully functional.
2. Server with valid `HANGAR_LICENSE_KEY` loads enterprise modules (auth, behavioral profiling, etc.).
3. Dashboard routes return 401 for unauthenticated requests regardless of license status.
4. License tier logged at startup: `license_tier=community|pro|enterprise`.
