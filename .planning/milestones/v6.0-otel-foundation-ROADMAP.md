# Milestone v6.0: OpenTelemetry Foundation

**Status:** IN PROGRESS
**Target:** 2026-04-15 (P0) / 2026-05-15 (P1)
**PyPI releases:** v0.13.0 (P0 items), v0.14.0 (P1 items)
**Phases:** 31-34
**Track:** Core (Python)
**Cross-cutting:** Runs in parallel with v7.0 (K8s Enforcement) and v8.0 (Behavioral Profiling)

## Overview

Establish MCP-aware OpenTelemetry semantic conventions and end-to-end trace context propagation as the governance telemetry contract between Hangar and partner backends (OpenLIT, Langfuse, Grafana, OTEL Collector). This milestone does not add a new user-facing feature -- it ensures that every security-relevant event emitted by Hangar carries stable, MCP-specific attributes that downstream tools can consume, correlate, and alert on.

This is an enabling milestone: all subsequent phases (K8s enforcement, behavioral profiling, identity propagation, semantic analysis) depend on the telemetry contract established here. P0 items land with v0.13.0. P1 items land with v0.14.0.

## Existing Foundation

| Area | What exists | What v6.0 adds |
|------|-------------|----------------|
| Tracing | `init_tracing()`, `trace_span()`, Langfuse adapter, basic OTLP export | Stable MCP-specific span attributes; governance context on every span |
| Metrics | 40+ Prometheus metrics, OTLP export | Security-relevant metric labels aligned with semantic conventions |
| Logging | structlog structured JSON | Audit signals exportable through OTLP logs |
| Trace propagation | Ad hoc correlation | W3C TraceContext propagated across agent -> Hangar -> provider boundaries |

**Phase dir:** `.planning/phases/32-trace-context-propagation/`

- [x] 32-02-PLAN.md -- Wire `inject_trace_context()` into `HttpLauncher` HTTP client -- inject traceparent into outbound request headers. Confirm StdioClient has no injection (correct by design).
