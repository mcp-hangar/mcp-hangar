## Requirements: MCP Hangar

**Defined:** 2026-02-15
**Core Value:** Reliable, observable MCP provider management with production-grade lifecycle control

## v0.9 Requirements

Requirements for Security Hardening milestone. Each maps to roadmap phases.

### Timing Attack Prevention

- [x] **TIME-01**: API key validation uses constant-time comparison (hmac.compare_digest) for all hash comparisons
- [x] **TIME-02**: All auth stores (InMemory, SQLite, Postgres, EventSourced) use constant-time lookup patterns
- [x] **TIME-03**: Tests verify timing characteristics remain within acceptable bounds

### Rate Limiter Hardening

- [x] **RATE-01**: Existing AuthRateLimiter passes comprehensive test suite (happy path + edge cases)
- [x] **RATE-02**: Rate limiter uses exponential backoff for lockout duration (not fixed 300s)
- [x] **RATE-03**: Rate limiter emits domain events for lockout/unlock transitions
- [x] **RATE-04**: Rate limiter cleanup handles edge cases (concurrent cleanup, timer drift)

### JWT Lifetime Enforcement

- [x] **JWT-01**: JWTAuthenticator enforces configurable max_token_lifetime (reject tokens where exp - iat > max_lifetime)
- [x] **JWT-02**: max_token_lifetime has sensible default (3600s) and is configurable via YAML/env
- [x] **JWT-03**: Rejected tokens produce clear error messages indicating lifetime violation

### Key Rotation

- [x] **KROT-01**: API key can be rotated via rotate_key() method producing new key while old remains valid during grace period
- [x] **KROT-02**: Grace period is configurable (default: 24h) after which old key is invalidated
- [x] **KROT-03**: Key rotation emits KeyRotated domain event with metadata (key_id, rotated_at, grace_until)
- [x] **KROT-04**: Rotated keys are tracked in auth stores (SQLite, Postgres, EventSourced)

## Future Requirements

### Security (deferred from v0.9)

- **SEC-F01**: API key IP binding (optional allowed_ips per key with CIDR support)
- **SEC-F02**: mTLS between Hangar and providers
- **SEC-F03**: Vault/HSM integration for key storage
- **SEC-F04**: Distributed rate limiting (Redis-backed for multi-instance)

## Out of Scope

| Feature | Reason |
|---------|--------|
| OIDC login flow | MCP Hangar is a resource server, not an IdP |
| API key IP binding | Deferred -- adds complexity without immediate threat |
| Vault/HSM integration | Production deployment concern, separate from auth hardening |
| mTLS for providers | Network-level concern, separate from auth layer |
