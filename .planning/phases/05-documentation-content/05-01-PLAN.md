Create `docs/reference/configuration.md` covering DOC-01 and DOC-02.

**Page structure:**

1. Title: `# Configuration Reference` with one-sentence description
2. Brief intro paragraph explaining config file location (`config.yaml` by default, overridden by `MCP_CONFIG` env var) and that env vars override YAML settings
3. Sections for each of the 13 YAML config sections (use `##` headings)
4. Final section: `## Environment Variables`

**For each config section (13 total):**

1. `## \`section_name\`` heading
2. Brief one-sentence description of what this section controls
3. YAML snippet showing common options with inline comments
4. Table with columns: Key | Type | Default | Range | Description
5. Use `!!! note` or `!!! warning` admonitions for gotchas (sparingly)

**The 13 sections to document (in this order):**

1. `providers` -- Provider definitions (mode, command, image, endpoint, idle_ttl_s, health_check_interval_s, max_consecutive_failures, env, volumes, build, resources, network/network_mode, read_only, user, args, description, tools dual format, auth, tls, http, max_concurrency). Note that container/podman normalize to docker. Note the `tools` dual format (list for predefined schemas vs dict for access policy with allow_list/deny_list).
2. `execution` -- System-wide concurrency (max_concurrency default 50, default_provider_concurrency default 10)
3. `discovery` -- Auto-discovery config (enabled, refresh_interval_s, auto_register, sources list, security sub-section, lifecycle sub-section). Include the sources[] entry keys and security/lifecycle sub-tables.
4. `retry` -- Retry policy (default_policy with max_attempts, backoff, initial_delay, max_delay, retry_on)
5. `event_store` -- Event persistence (enabled, driver sqlite/memory, path)
6. `knowledge_base` -- Knowledge base storage (enabled, dsn, pool_size, cache_ttl_s)
7. `logging` -- Log configuration (level default INFO, json_format default false, file)
8. `health` -- Health check config (enabled, interval_s)
9. `observability` -- Tracing and Langfuse sub-sections. Include tracing keys (enabled, otlp_endpoint, service_name, jaeger_host, jaeger_port, console_export) and langfuse keys (enabled default false, public_key, secret_key with env var interpolation note, host, sample_rate, scrub_inputs, scrub_outputs)
10. `auth` -- Authentication (enabled, allow_anonymous, api_key sub, oidc sub, opa sub, storage, rate_limit, role_assignments)
11. `config_reload` -- Hot reload (enabled, use_watchdog, interval_s)
12. `batch` -- Batch invocation limits (max_calls default 100, max_concurrency default 50, default_timeout default 60, max_timeout default 300, max_response_size_bytes default 10MB, max_total_response_size_bytes default 50MB)
13. `groups` -- Group configuration within providers section (mode="group", strategy, min_healthy, auto_start, description, health sub, circuit_breaker sub, tools, members). Include MemberWeight 1-100 and MemberPriority 1-100 ranges.

**Environment Variables section (at the end):**

Group by category with sub-headings (`###`):

- Server / CLI: MCP_CONFIG, MCP_MODE, MCP_HTTP_HOST, MCP_HTTP_PORT, MCP_LOG_LEVEL, MCP_JSON_LOGS
- Security / Runtime: MCP_RATE_LIMIT_RPS, MCP_RATE_LIMIT_BURST, MCP_ALLOW_ABSOLUTE_PATHS
- Persistence: MCP_PERSISTENCE_ENABLED, MCP_DATABASE_PATH, MCP_DATABASE_WAL, MCP_AUTO_RECOVER
- Observability / Tracing: MCP_TRACING_ENABLED, MCP_TRACING_CONSOLE, MCP_ENVIRONMENT, OTEL_EXPORTER_OTLP_ENDPOINT, OTEL_SERVICE_NAME, JAEGER_HOST, JAEGER_PORT
- Langfuse: MCP_LANGFUSE_ENABLED, LANGFUSE_PUBLIC_KEY, LANGFUSE_SECRET_KEY, LANGFUSE_HOST, MCP_LANGFUSE_SAMPLE_RATE, MCP_LANGFUSE_SCRUB_INPUTS, MCP_LANGFUSE_SCRUB_OUTPUTS
- Container Runtime: MCP_CONTAINER_RUNTIME, MCP_CI_RELAX_VOLUME_PERMS, MCP_CONTAINER_INHERIT_STDERR
- Auth: MCP_JWT_MAX_TOKEN_LIFETIME

Each category as a table with columns: Variable | Default | Description

Include a `!!! note` admonition about legacy `HANGAR_*` prefix alternatives (HANGAR_LANGFUSE_ENABLED, HANGAR_LANGFUSE_SAMPLE_RATE, HANGAR_LANGFUSE_SCRUB_INPUTS, HANGAR_LANGFUSE_SCRUB_OUTPUTS) noting they exist for backward compatibility but `MCP_*` is canonical.

**Style rules:**

- Follow existing reference page style from cli.md and hot-reload.md
- Direct technical tone, minimal prose between tables
- No emoji
- Use Python 3.11+ type hint syntax in type columns (e.g., `list[str]` not `List[str]`)
- Use `--` for "no default" or "not applicable" in table cells

All content data is in 05-RESEARCH.md sections "Complete Configuration Schema" and "Complete Environment Variables".

Create `docs/reference/tools.md` covering DOC-03 and DOC-04.

**Page structure:**

1. Title: `# MCP Tools Reference` with one-sentence description
2. Brief intro explaining these are MCP protocol tools exposed by Hangar, callable by any MCP client
3. Quick-reference summary table listing all 22 tools (columns: Tool | Category | Description | Side Effects) with anchor links to each tool card
4. Tool cards organized by 7 categories (use `##` for categories, `###` for individual tools)

**Quick-reference summary table:**

All 22 tools in a single table at the top of the page. Each tool name is an anchor link to its detailed card below. The "Side Effects" column shows "None (read-only)" or a brief description.

**7 categories with their tools:**

1. `## Lifecycle` (5 tools): hangar_list, hangar_start, hangar_stop, hangar_status, hangar_reload_config
2. `## Hot-Loading` (2 tools): hangar_load, hangar_unload
3. `## Provider` (3 tools): hangar_tools, hangar_details, hangar_warm
4. `## Health` (2 tools): hangar_health, hangar_metrics
5. `## Discovery` (5 tools): hangar_discover, hangar_discovered, hangar_quarantine, hangar_approve, hangar_sources
6. `## Groups` (2 tools): hangar_group_list, hangar_group_rebalance
7. `## Batch and Continuation` (3 tools): hangar_call, hangar_fetch_continuation, hangar_delete_continuation

**Each tool card structure (use `### \`tool_name\`` with `{#tool_name}` anchor):**

1. One-sentence description (rewrite from docstrings for human audience -- do NOT copy CHOOSE THIS/SKIP THIS guidance)
2. **Parameters** table: Parameter | Type | Default | Range | Description (omit Range column if tool has no ranged params)
3. **Side Effects** line: "None (read-only)" or brief list of state changes
4. **Returns** table: Field | Type | Description. For tools with multiple return shapes (success/error/etc), show each as a labeled sub-section or use prose to explain variants.
5. **Example** code block: One JSON request/response pair. Use `json` fenced code block with `// Request` and `// Response` comments.

**Important notes:**

- Do NOT copy CHOOSE THIS/SKIP THIS docstring guidance -- these are for LLM tool selection, not human readers
- Use `list[object]` and `str | None` Python 3.11+ syntax in type columns
- For tools with multiple return shapes (like hangar_load with 6 possible statuses), document the primary success case in the returns table and describe variants in a brief paragraph
- Mark async tools with "(async)" after the tool name heading
- Note error conditions inline (e.g., `ValueError("unknown_provider: <id>")`)

All content data is in 05-RESEARCH.md section "Complete MCP Tool Inventory".
