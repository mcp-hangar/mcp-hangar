Create `docs/guides/PROVIDER_GROUPS.md` covering DOC-05 and DOC-06.

**Page structure (follow BATCH_INVOCATIONS.md pattern: overview, reference, examples, limits):**

1. Title: `# Provider Groups` with one-sentence description
2. `## Overview` -- What groups are, when to use them (high availability, load distribution, failover). Explain that groups appear as a single provider to MCP clients. Mention the 4 group states (inactive, partial, healthy, degraded) briefly.
3. `## Configuration` -- Basic group YAML structure showing a minimal group config with inline comments. Explain `mode: group` triggers group behavior. Show the base keys: strategy, min_healthy, auto_start, description, members.
4. `## Load Balancing Strategies` -- One `###` sub-section per strategy (5 total). Each sub-section: brief behavioral description + YAML config example + notes on when to choose this strategy.
5. `## Health Policy` -- How member health tracking works. Document unhealthy_threshold (default 2) and healthy_threshold (default 1). Explain the removal/re-entry flow: consecutive failures >= unhealthy_threshold removes from rotation; consecutive successes >= healthy_threshold AND provider state == READY re-adds.
6. `## Circuit Breaker` -- Group-level circuit breaker. Document failure_threshold (default 10), reset_timeout_s (default 60.0). Explain states: CLOSED (normal) -> OPEN (all requests rejected) -> auto-reset after timeout -> CLOSED. Note that `hangar_group_rebalance` resets the circuit breaker.
7. `## Tool Access Filtering` -- Three-level policy hierarchy (provider-level, group-level, member-level). Explain fnmatch glob pattern matching. Show YAML example with allow_list/deny_list at each level. Explain resolution: allow_list takes precedence if set; deny_list only applies if allow_list empty; empty lists = all tools visible.

**5 Strategy sub-sections:**

1. `### Round Robin` -- Sequential cycling through healthy members. No weight/priority. YAML example with 2+ members.
2. `### Weighted Round Robin` -- Nginx smooth weighted algorithm, higher weight = more requests. Uses `weight` (1-100, default 50). YAML example showing members with different weights.
3. `### Least Connections` -- Selects member with oldest `last_selected_at` timestamp (effectively least recently used). No weight/priority. YAML example.
4. `### Random` -- Weighted random selection from healthy members. Uses `weight` as probability weight. YAML example.
5. `### Priority` -- Selects lowest priority number. Primary/backup pattern. Uses `priority` (1-100, lower = higher priority). YAML example showing primary (priority: 1) and backup (priority: 50) members.

**Group states reference (include as a table in Overview or as its own brief section):**

| State | Condition | Accepts Requests |
|-------|-----------|------------------|
| inactive | 0 healthy members | No |
| partial | healthy < min_healthy | Yes (if circuit closed and healthy >= 1) |
| healthy | healthy >= min_healthy | Yes |
| degraded | Circuit breaker open | No |

**Style rules:**

- Follow existing guide style from BATCH_INVOCATIONS.md (overview + reference + examples + limits)
- Direct technical tone, no tutorial hand-holding
- No emoji
- YAML code blocks for all config examples
- Cross-reference to Configuration Reference page for full YAML schema details where appropriate
