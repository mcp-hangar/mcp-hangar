<objective>
Integrate capabilities-aware network isolation into Docker and container provider launchers. When a provider declares capabilities.network, use the egress list to determine network mode: empty egress = `--network none`, any egress = bridge network.

Purpose: Provides Docker-mode enforcement equivalent to K8s NetworkPolicy (coarser grain -- binary deny/allow). Non-K8s environments get at least isolation-by-default when capabilities are declared.

Output: Modified Docker/container launchers with capabilities-driven network mode + unit tests.
</objective>

3. Replace the network logic block (line 147-148) with capabilities-aware logic:
```python
# Network isolation: capabilities override enable_network flag
network_deny = False
if capabilities is not None and capabilities.network is not None:
    # Capabilities-driven: deny if no egress declared
    if not capabilities.network.egress:
        network_deny = True
    # else: egress declared -> allow network (bridge)
elif not self._enable_network:
    # Legacy fallback: use constructor flag
    network_deny = True

if network_deny:
    cmd.extend(["--network", "none"])
```

4. Update the `launch()` method to pass capabilities through to `_build_docker_command`. The `launch()` method currently calls `_build_docker_command(image, env)`. Add a `capabilities` parameter to `launch()`:
```python
def launch(
    self,
    image: str,
    env: dict[str, str] | None = None,
    capabilities: ProviderCapabilities | None = None,
) -> StdioClient:
```
And pass it through: `cmd = self._build_docker_command(image, env, capabilities=capabilities)`

Note: The `launch()` method signature extends the base class. Since Python allows adding optional kwargs to overridden methods, this is backward compatible. Callers that do not pass `capabilities` get the existing behavior.
